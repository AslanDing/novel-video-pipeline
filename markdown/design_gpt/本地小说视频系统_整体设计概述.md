# 本地小说视频系统：整体架构与设计方案概述

## 一、 系统定位与目标
本系统是一个专门针对**“小说到视频（Novel-to-Video）”**自动化生产流程而构建的多阶段内容生产平台。它运行在本地单张 24GB VRAM GPU 环境下，通过编排调度不同的AI模型（大语言模型、语音合成、文生图、图生视频等）实现“构思-小说-分镜脚本-旁白与对白-视频镜头-最终成片”的全链路闭环。

系统的核心设计思想在于：**拒绝简单的黑盒串行脚本，拥抱模块化、版本化、局部重跑和中间产物一等公民化。** 用户可以在任何中间节点进行修改、审批或替换（比如修改一句台词、替换某一定格、切换某人物的预设特征等）。

---

## 二、 核心架构设计

系统采用了清晰的**分层架构模式**，隔离业务状态、编排逻辑及底层模型调用，以此解决 GPU 资源受限及长作业容易中断等难题。

### 2.1 分层模型
1. **UI / API 层**：提供Web UI或者前端任务中心，处理用户发起的上传构思、调整分镜、手动修正及局部重跑指令。将请求抽象为长任务指令（Command Event）。
2. **Orchestrator（编排引擎层）**：系统的大脑。负责解析任务工作流（Workflow），将其拆分为子任务 DAG（有向无环图），驱动状态机，并处理复杂的重跑、回滚、补偿与产物版本控制。
3. **Domain Layer（领域业务层）**：定义核心实体与生命周期如 Project（项目）、Episode（章节）、Scene（场景）、Shot（镜头）以及相关的 Voice/Visual Profile（声音与视觉锚点）和 Artifact（模型产物与版本资产）。
4. **Worker Layer（原子任务执行层）**：无状态的工作节点。主要包括：
   - **LLM Worker**：撰写大纲/正文、拆分剧本、改写 prompt 和质量检查。
   - **TTS Worker**：进行语音声学合成、台词切分和谐音修正。
   - **Image Worker**：生成人物/背景概念图及镜头关键帧。
   - **I2V Worker**：驱动关键帧生成具有动态位移的视频镜头。
   - **QC / Compose Worker**：全自动质量检查（ASR等）及最终时间线 FFmpeg 拼装输出。
5. **Infra Layer（基础设施层）**：本地持久化存储、Artifact 缓存、版本配置控制及至关重要的 **GPU Lock（显存调度仲裁）**。

### 2.2 软硬件条件下的 GPU 调度策略
单张 24GB 卡无法并发承载所有的生成流水线（文本推理+图生图+视频扩散），因此采用**细粒度 GPU 锁机制**：
- 任务受资源占用分级：重（I2V, 大型文生图）、中（TTS, LLM）、轻（FFmpeg合成，文本组装处理）。
- 重任务完全独占并带有降级补偿策略（如OOM时缩减视频帧率和分辨率）。
- 绝不使用常驻显存策略去空等，用任务队列激活对应的 Worker 竞争 GPU 使用权。

## 三、 四大关键子系统概述

### 3.1 LLM 文本与脚本子系统
此系统不是单一的大段 prompt 推理，而是按照“Planner -> Bible -> Outline -> Novel -> Script -> Prompt Rewrite -> Context Checker”构建的多阶段**Prompt Chain**。
- **功能职责**：根据构思生成整个世界观和角色设定（Story Bible），细化章节正文，再由正文转化为镜头级 JSONL 分镜脚本。
- **质量保证**：采用强 Schema（JSON 契约），引入文本长度验证与长背景上下文摘要（Context Pack）进行隔离。

### 3.2 TTS 与音频流水线子系统
- **功能职责**：将拆分的脚本生成角色旁白/对白音频。使用 Voice Profile 定制角色音色资源。
- **细粒度追踪**：可切分长难句或重试单句的发音；保留单句级别的 Timeline（起止毫秒时间戳），为最终复合字幕（SRT/ASS）提供绝对时间点源。
- **错误发现**：可以通过反向调用 ASR 分析音频是否有漏字、读错。

### 3.3 视觉关键帧子系统 (Image Worker)
系统视**角色一致性为第一要求**。
- **视觉锚点(Visual Profile)**：建立在文学描述之外的图像固定参考池（Reference-first），包括服饰、身体特征及常用 Seed 策略。
- **控制输入**：不让用户手写提示词，由大模型负责将剧情意图改写为扩散模型能听懂的正负提示词并搭载 ControlNet（结构、姿势约束）。
- **人工干预闭环**：关键帧可以被人工选择、舍补和手动上传覆盖，作为下一阶段 I2V 视频生成的基础底图。

### 3.4 I2V 短视频成片与合成 QC 子系统
- **I2V 镜头生成**：基于前面过审的图片关键帧和运动意图（Motion prompt），产出短片段（3-5s）的微动态视频。支持镜头级降级（当显卡跑不过去时从720p妥协到480p或者使用少步数候选）。
- **智能视频合成（Timeline / Compose）**：不是单纯的 concat，而是基于声音段长、对齐字幕线为主基线来容差修剪或微弱缩放（Freeze/Speed）各个散装短视频。提供集级全流程输出及最终报错汇报。

## 四、 Artifact 与 局部重跑机制 (核心设计亮点)

在单机全周期链路中，任务极其容易因为生成失控毁掉整章的产出。故设计了强大的内部追溯机制：
1. **Immutable Artifacts (产物不可变原则)**：所有的请求阶段输出的均是一个唯一指向的 Artifact 对象，生成后不可修改。用户/重试产生的任何变更均创建新 Version。
2. **Superseded 状态传播**：一旦修改了父级资产（例如将第4个镜头的关键帧手动换了一张图），编排引擎捕捉事件后就会沿着 DAG 让下游衍生的“前版本镜头”直接失效变色，将其加入到 Ready 队列以排期更新。
3. **安全的人工协作**：赋予用户强大的干预能力而不破坏整体运行进程。系统完全可以在后台继续推进合法镜头，同时把错误帧留给人工。此机制为项目长时间平稳落地和高质量验收提供了护城河。

## 五、 开发规划建议

项目包含众多 AI 模型集成及繁重的工程协调负担，建议依据如下开发路径逐步攻坚：
1. **M1 骨架建设**：搭建 Orchestrator 引擎、数据库与统一接口 Schema、建立 GPU Lock。
2. **M2-M4 AI 模块原子化验证**：依次点亮 LLM 脚本拆分 -> TTS 音色生成流 -> 关键帧图库组建。不追求 UI 交互，先追求 JSON 到 JSON 的通畅。
3. **M5-M6 合成与工作流闭环**：拼装视频产线，实现重试和失效传导链条。搭建前端面板暴露核心管控点，开启完整集别的本地压测。
