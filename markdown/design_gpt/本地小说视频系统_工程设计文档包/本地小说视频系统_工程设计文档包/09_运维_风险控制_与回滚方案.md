# 09 运维、风险控制与回滚方案

## 9.1 运维目标

虽然系统是本地单机版，但仍然需要“可运维性”。  
最少要解决：

- 任务崩溃后恢复
- 显存占用失控后的回收
- 产物损坏后的回滚
- 配置变更后的可追溯
- 磁盘空间增长控制

---

## 9.2 环境管理

建议使用：

- Python 虚拟环境或 Conda
- 每个重模型独立环境（可选）
- `.env` 管理路径和端口
- `config.yaml` 管理全局默认参数

### 示例环境变量
```bash
PROJECT_ROOT=/data/novel_video
DB_URL=sqlite:///data/app.db
LLM_API_BASE=http://127.0.0.1:8001/v1
COMFYUI_BASE=http://127.0.0.1:8188
GPU_LOCK_PATH=/tmp/novel_video_gpu.lock
```

---

## 9.3 日志与监控

### 必要日志
- orchestrator 主日志
- 每类 worker 独立日志
- 项目级执行报告
- 失败任务错误栈

### 建议指标
- 每阶段平均耗时
- GPU job 等待时间
- 失败率
- 重试次数
- 磁盘占用
- 缓存命中率

本地版可以先把指标落 JSON 或 SQLite，不必一开始就接 Prometheus。

## 9.4 风险清单

## 风险 1：GPU OOM
触发点：
- 高分辨率图像生成
- 长时长 I2V
- LLM 上下文过长

应对：
- 统一 GPU lock
- 自动降级 preset
- 失败后释放显存再重试

## 风险 2：角色漂移
触发点：
- 参考图不足
- prompt 变动过大
- 镜头跨度太大

应对：
- 强制使用 reference pack
- 关键角色固定 visual anchors
- 重要场景人工审图

## 风险 3：脚本不可生产
触发点：
- 镜头过长
- 对白过密
- 场景变化过快

应对：
- script 生成阶段加入规则约束
- 添加 script lint
- 自动拆分过长 shot

## 风险 4：TTS 发音错误
触发点：
- 专有名词
- 多音字
- 标点异常

应对：
- text normalization
- pronunciation override
- ASR 回查

## 风险 5：文件污染或覆盖
触发点：
- 直接覆写 active 版本
- 重跑逻辑不安全

应对：
- 新版本永远写新文件
- 通过 metadata 切换 active
- 旧版本保留到归档

## 风险 6：磁盘空间暴涨
触发点：
- 多次视频重跑
- 保留大量预览中间件

应对：
- 低优先级自动归档
- 清理 deprecated 视频
- 保留 active + 最近 N 版

## 9.5 回滚策略

### 回滚粒度
- task 级
- artifact 级
- episode 级
- project config 级

### 回滚原则
1. 不删除历史版本
2. 只切换 active 指针
3. 回滚必须写审计记录

### 示例：回滚某镜头视频
1. 查询 `shot_id = S01_SH03`
2. 找到上一版 `video artifact v2`
3. 将 v2 标记为 `active`
4. 当前 v3 标记为 `deprecated`
5. 重生成 compose 任务

## 9.6 备份策略

### 必备备份
- 数据库
- 项目配置
- active artifacts 索引

### 可选备份
- 最终成片
- 关键角色参考图
- 已审核通过的关键帧

建议：
- 每个项目导出一份 manifest
- 每晚备份 SQLite + active metadata

## 9.7 故障恢复手册

### 场景 A：程序异常退出
处理：
1. 重启 orchestrator
2. 扫描 `running` 任务
3. 检查输出是否存在
4. 将未完成任务改为 `queued`
5. 从最近安全点恢复

### 场景 B：GPU 卡死
处理：
1. 杀死对应 worker
2. 清理显存
3. 标记当前 run 为 `retryable_failed`
4. 重新排队

### 场景 C：文件缺失
处理：
1. 校验 artifact file path
2. 若文件不存在，artifact 标为 `invalid`
3. 自动回滚到上一 active 版本或触发重跑

### 场景 D：数据库损坏
处理：
1. 停止写入
2. 载入最近备份
3. 根据文件 manifest 重建缺失索引

---

## 9.8 安全与隔离

本地版本虽然单用户，仍建议：
- worker 只访问受控目录
- 禁止任意路径写入
- 输入文件做扩展名与大小检查
- 内部接口不暴露公网
- 命令执行层避免拼接 shell 字符串

---

## 9.9 发布与升级策略

### 升级前
- 备份数据库
- 备份全局配置
- 记录当前模型版本
- 记录当前 prompt 模板版本

### 升级后
- 运行 smoke test
- 对样例项目做最小闭环测试
- 确认 active artifact 索引未损坏

### 禁止直接升级的情况
- 数据 schema 未迁移
- worker 接口不兼容
- prompt 模板发生破坏性变更但无版本号

---

## 9.10 运营建议（即便是本地工具）

建议每个项目生成：
- `manifest.json`
- `release_note.md`
- `asset_index.csv`

这样未来回看素材和复用资产时，成本会低很多。

---

## 9.11 最终建议

这套系统真正的护城河，不是某一个模型，而是：

- 任务拆分是否合理
- 数据结构是否清晰
- worker 是否解耦
- 局部重跑是否可靠
- 中间资产是否可复用

只要这些工程底座搭好，后面换模型、换预设、换风格，都会容易很多。
