# 01 项目总览与目标

## 1.1 项目背景

目标系统不是单一“超大一体化模型”的演示，而是一条面向生产的本地流水线。  
输入是一段小说构思，输出是一个包含旁白、对白、关键帧、动态镜头和字幕的短视频成片。

在单张 24GB VRAM GPU 上，最重要的不是追求一次性生成高时长、高分辨率的视频，而是：

- 生成过程稳定
- 角色与场景一致性可控
- 失败可局部重跑
- 中间资产可复用
- 成片质量可逐步迭代

因此本项目采用**分阶段生成 + 中间资产持久化 + 可恢复任务编排**的工程路线。

---

## 1.2 核心目标

### 业务目标
1. 用户输入一个小说构思
2. 系统自动生成故事设定、章节正文和分镜脚本
3. 系统自动生成旁白、对白和字幕
4. 系统自动生成关键帧和短视频镜头
5. 系统自动拼接为成片
6. 用户可以只重做局部，而不是全量重跑

### 工程目标
1. 完全本地可运行
2. 单 GPU 24GB 显存内可落地
3. 所有中间产物标准化存储
4. 所有任务状态可追踪
5. 所有失败节点可恢复
6. 支持扩展为多章节、多项目批量生产

---

## 1.3 非目标

以下内容不属于 v1 范围：

- 单句构思直接生成 10 分钟以上高质量长视频
- 全角色全程精准口型同步
- 实时交互式预览
- 多机分布式渲染
- 面向公网多租户 SaaS 版本

这些能力可以在 v2/v3 再扩展。

---

## 1.4 成功标准

### 功能成功标准
- 能从一个构思生成至少一章小说正文
- 能从该章生成结构化分镜脚本
- 能生成旁白和至少两个角色的对白
- 能生成 8–20 个镜头的关键帧与视频片段
- 能导出一条带音频与字幕的完整短片

### 工程成功标准
- 任意镜头失败后可单镜头重跑
- 单个镜头重跑不会破坏其他资产
- 任意阶段的产物均有版本号和元数据
- 系统崩溃重启后可从最近安全点恢复
- 日志可定位失败点与输入输出关系

---

## 1.5 总体设计原则

### 原则 1：短镜头优先
视频主线单位不是“整章”，而是“shot”。  
每个 shot 推荐长度控制在 2–6 秒。

### 原则 2：I2V 优先于纯 T2V
对 24GB 单卡，图生视频比纯文生视频更稳，尤其适合小说这种强调角色和构图连续性的场景。

### 原则 3：中间资产显式化
每个阶段必须输出结构化产物，而不是只依赖内存态。  
例如：
- `story_bible.json`
- `ep01_script.jsonl`
- `audio_line_xxx.wav`
- `keyframe_shot_xxx.png`
- `shot_xxx.mp4`

### 原则 4：接口驱动
模块之间通过 JSON 和文件契约解耦，而不是通过隐式变量共享。

### 原则 5：失败可恢复
所有长耗时任务必须具备：
- 幂等性
- 重试策略
- 超时控制
- 中断恢复能力

---

## 1.6 产物层级

建议按以下层级理解整个系统：

### 层级 1：故事资产
- 构思
- 故事圣经
- 角色圣经
- 章节大纲
- 正文

### 层级 2：生产脚本资产
- 分镜脚本
- 台词脚本
- 镜头视觉提示
- TTS 指令
- I2V 提示

### 层级 3：媒体资产
- 声音
- 关键帧
- 镜头视频
- 字幕
- 最终成片

### 层级 4：工程资产
- 任务日志
- 元数据
- 版本记录
- QA 结果
- 缓存与索引

---

## 1.7 建议的 v1 输出形态

v1 不追求“完全动画电影”，而是优先做：

- 动态漫画
- 旁白驱动的分镜短片
- 有对白的章节演绎
- 角色驱动的剧情片段

这条路线对 24GB 单卡最友好，也最能快速闭环。

---

## 1.8 项目角色分工建议

### 1）产品 / 需求负责人
负责：
- 故事类型范围
- 用户输入形式
- 成片标准
- 里程碑定义

### 2）后端 / 流程工程师
负责：
- Orchestrator
- 状态机
- 任务队列
- API 层
- 文件管理
- 缓存与恢复

### 3）算法 / 模型工程师
负责：
- LLM prompt 设计
- TTS 选型与调优
- 图像与视频工作流调优
- LoRA / 参考图一致性策略

### 4）工具链 / 运维
负责：
- 本地部署脚本
- 环境管理
- GPU 占用监控
- 日志与备份

---

## 1.9 最终建议

先把系统做成“可重复生产一个 3–10 分钟章节视频”的稳定工具。  
等主链稳定后，再做：
- 角色口型同步
- 自动 BGM
- 镜头转场库
- 批量章节处理
- 风格化 LoRA
