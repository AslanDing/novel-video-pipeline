# 本地小说视频系统：完整开发流程与方法论

为了保证这个系统在复杂多变（尤其是各种不同的大模型集成与硬件受限环境）条件下的稳定产出，依据软件工程的经典原理（结合敏捷迭代和瀑布式底层规划），我们为“本地小说到视频生成系统”制定了以下完整的开发流程指引。

---

## 总体开发策略：基于原型的增量迭代
由于 AI 系统的极高**非确定性**（Non-deterministic），我们不能等到所有组件都写完才做联调测试。我们采取**核心链路优先，逐步挂载高级特性**的开发模型，以模块闭环作为每个开发里程碑的标志。

---

## 一、 第零阶段：需求分析与环境基线确立 (Requirements & Baseline)
**目标**：明确系统的能力边界与软硬件前提条件。

1. **核心约束明确**：
   - 硬件红线：单卡 24GB VRAM GPU 能够独立运行完毕整个流程，绝不出现显存爆炸（OOM）的停机断电事故。
   - 产出预期：优先保证生成短视频序列流畅拼接，而不是动辄去生成复杂的动作长镜头。
2. **术语系统（Domain Dictionary）对齐**：
   - 确定并统一上下游对 Project、Episode、Scene、Shot、Artifact 的语义理解，写入文档字典。
3. **技术栈选型与验证**：
   - 比如 FastAPI + SQLAlchemy (后端)、Vue/React (任务中心UI)、本地缓存队列（SQLite/Celery）。
   - 实机跑通 vLLM（文本）、Fish Speech / CosyVoice（TTS）、图文生成及视频生成模型的 Hello World 级测试，评估单次推理的基础显存开销。

---

## 二、 第一阶段：架构设计与防腐层规划 (Architecture & Design)
**目标**：拆分系统边界，定义通信契约。

1. **数据模型与 Schema 定义**：
   - 设计数据库表结构（如 Projects, Shots, Artifacts 追踪表）。
   - 定义标准的任务输入/输出 JSON 契约（确保各子模块只认标准的 JSON，而不互相深度侵入代码）。
2. **编排与状态机编绘**：
   - 定义任务的有向无环图 (DAG) 与重跑失效规则，规划系统状态应该如何正确回滚（比如“重新生成了某一关键帧，下游对应的视频自动状态变灰”）。
3. **资源管理设计**：
   - 规划并设计统一的 GPU 资源锁（GPU Lock），区分轻、中、重任务，保证显卡的串行调度策略落实到设计稿。

---

## 三、 第二阶段：核心骨架基础设施搭建 (Core Infra Build / Milestone 1)
**目标**：拉起空跑的框架骨架。

1. **项目骨架与构建脚本**：
   - 规范多包（Monorepo 或清晰目录划分）的代码仓，搭建 config 系统，统一日志中心（Logger）。
2. **Orchestrator（编排引擎）引擎与任务队列起步**：
   - 完成任务入队、调度和出队等调度基础逻辑。
   - 搭建 Fake Worker 进行测试（模拟 AI 推理睡眠几秒然后返回 JSON），跑通整个 DAG。
3. **Artifact 版本管理持久化**：
   - 将每一次空跑产生的“模拟资源”录入数据库及文件系统，验证是否能正确形成版本链。

---

## 四、 第三阶段：AI 能力子流程集成 (AI Pipeline Integration)
**目标**：将真实的 AI 生成能力按“从上游到下游”的顺序分层接入，并且达成每一个子系统的闭环。

### 4.1 文本与脚本生成流 (M2)
- 联调并接入 LLM Worker，验证 Story Bible -> Outline -> Novel -> JSONL 分镜脚本输出能力。
- 添加 Prompts 的模块化模板管控。
### 4.2 音频与时间线生成流 (M3)
- 联调 TTS Worker 进行文本到语音的转换，加入切句缓冲、音色提取与配置功能。
- 从音频提取对齐关键信息（Timing），输出粗略字幕文件。
### 4.3 视觉关键帧闭环 (M4)
- 引入 Image Worker，搭载角色参照与负面约束（ControlNet/LoRA）。
- 建立关键候选集（Candidate）的排序评价小引擎。
### 4.4 视频与合并流闭环 (M5)
- 测试 I2V 推理。
- 搭建 FFmpeg 组装管线进行最终短视频合并与字幕烧录输出。

---

## 五、 第四阶段：用户交互控制与可视化 (UI & Manual Interventions)
**目标**：将不可控变为可控，引入“人机协作”。

1. **Task Center 前端面板开发**：
   - 搭建任务监测的控制大屏，能实时看到每个 Shot 的进度流转。
2. **人工审查与介入模块（Human-in-the-loop）**：
   - 支持图形化页面中手动替换/上传一张自选图片覆盖 AI 生成的关键帧。
   - 实现重试按钮发起指定局部范围的重跑命令。
3. **Debug 面板开发**：
   - 对外展露错误日志、执行耗时 metric 和当前生效的 Prompt 源字符串，方便追踪问题。

---

## 六、 第五阶段：集成测试与质量保障 (Testing & Quality Assurance)
**目标**：提高稳健性，实现真正的自动化交付。

1. **自我检查 QC 模块联调**：
   - 检查生成视频是否黑屏、音频音量是否异常、合成长短是否撕裂。遇到报错由 Orchestrator 自动发起降级或重跑。
2. **基线集成测试 (E2E Test)**：
   - 录入三个固定标准测试案列（比如 5个镜头的都市短故事、10个镜头奇幻故事），跑全局全流程。
3. **性能及极限测试**：
   - 触发 OOM、制造网络断连或硬盘满载报错等异常流的故障演练，验证系统的错误码捕获功能及脏数据清理能力。

---

## 软件工程实践与规范要求 (Engineering Practices)

1. **Git 工作流策略**
   - 采用标准 `feature/*` -> `develop` -> `main` 机制分支演进。
   - API 和数据模型的改动必须前置发起 PR（Pull Request）评审。
2. **DoD (Definition of Done) 标准**
   - 新接入的任何模型或处理 Worker：
     (1) 明确了入参校验和标准的出参 Schema；
     (2) 内部报错被 try-catch 并返回标准的错误码；
     (3) 提供了独立的运行 README 或者最简单的本地直接调用的 mock 测试脚本。
3. **配置管理分离**
   - AI 的 Prompt 工程不要写死在代码里，将其模板化存入版本库或配置系统中。
4. **渐进式重构**
   - 前期可用简单的本地 SQLite 和本机简单线程池完成任务队列，项目平稳后如有大量规模生成诉求，再横向迁移至 PostgreSQL + Redis + Celery 等专业分布方案，保证底层接口透明。
