# 21_可观测性_日志_指标_与调试面板设计

## 1. 目标

创意生成系统的调试难度远高于普通 CRUD 系统。你需要的不只是错误日志，还需要知道：

- 哪一步用了哪个 prompt
- 哪个版本的参考图被选中了
- 哪次 TTS 用了哪个 voice profile
- 某个镜头为什么失败
- 为什么同样的输入这次效果不同

---

## 2. 可观测性四层

1. **事件层**：发生了什么
2. **任务层**：哪个任务执行了什么
3. **Artifact 层**：产生了哪些版本
4. **质量层**：输出效果怎么样

---

## 3. 日志设计

### 3.1 结构化日志字段
- `timestamp`
- `level`
- `project_id`
- `episode_id`
- `scope_id`
- `task_id`
- `attempt_id`
- `worker`
- `event_type`
- `message`
- `error_code`
- `latency_ms`

### 3.2 不建议只写自由文本
所有关键日志都应 JSON 化，便于聚合和过滤。

---

## 4. 指标设计

### 4.1 系统指标
- CPU 使用率
- GPU 利用率
- VRAM 峰值
- 磁盘使用量
- 队列长度

### 4.2 任务指标
- 平均耗时
- 成功率
- 重试率
- 失败原因分布

### 4.3 质量指标
- TTS ASR 一致率
- 关键帧自动评分均值
- I2V QC 通过率
- 镜头重跑率

---

## 5. 调试面板

### 页面 1：项目总览
- 项目状态
- 当前章节进度
- 最近失败任务
- 磁盘占用

### 页面 2：任务树
- 按 episode / scene / shot 展开
- 查看状态、耗时、失败原因
- 支持重跑

### 页面 3：Artifact 浏览器
- 查看同一 artifact 的多个版本
- 对比差异
- 切换 current version

### 页面 4：运行时详情
- effective config
- prompt 版本
- 参考资产列表
- GPU metrics

---

## 6. Trace 能力

建议给每个顶层请求分配 `trace_id`，并贯穿：
- API
- Orchestrator
- Worker
- Final artifact

这样可以从导出视频反查整条生成链。

---

## 7. 关键告警

- GPU lock 超时
- 队列堆积
- 连续 OOM
- 某个 worker 连续崩溃
- 磁盘占用超过阈值
- QC 通过率突然下降

---

## 8. 实现建议

- v1 可用数据库事件表 + JSON 日志
- v2 接 Prometheus/Grafana
- UI 中优先做项目时间线和任务树，不必一开始做很重的监控平台

---

## 9. 评审 checklist

- 是否能从最终视频追溯到每一步
- 是否有 task / attempt 级日志
- 是否有质量指标
- 是否能对 artifact 做版本比较
- 是否能定位失败的最小范围
