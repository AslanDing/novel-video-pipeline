# 本地小说到视频生成系统完整方案书（24GB VRAM 单卡版）

版本：v1.0  
日期：2026-03-29  
适用对象：希望在本地、单机、24GB VRAM 左右环境下，把“小说构思”自动转换为“小说正文 + 多人配音 + 关键帧 + 图生视频 + 合成成片”的开发者与独立团队。

---

## 1. 项目目标

本方案的目标不是做一个“单个超大模型直接从一句话生成长视频”的演示系统，而是搭建一条**稳定、可控、可迭代、完全本地可运行**的生产流水线：

1. 用户输入一个小说构思。
2. LLM 生成小说圣经、角色设定、章节大纲、章节正文。
3. LLM 把正文改写成适合音视频生产的镜头脚本。
4. 本地 TTS 生成旁白与多人对白。
5. 文生图系统生成角色设定图、场景图、剧情关键帧。
6. 图生视频模型把关键帧转换为短视频镜头。
7. FFmpeg 合成音频、视频、字幕，导出最终成片。
8. 系统支持迭代修正：重写某个镜头、重生成某句对白、替换某个角色形象，而不是全量重跑。

---

## 2. 设计原则

### 2.1 第一原则：在 24GB 单卡上真实可跑
系统必须服务于你的真实硬件，而不是依赖多卡或云端。  
因此模型选择、调度方式、分辨率、时长设定都要围绕“单 GPU 24GB 可落地”来设计。

### 2.2 第二原则：先可控，再极致画质
对小说视频化来说，最重要的不是单帧天花板画质，而是：
- 角色是否稳定
- 镜头是否可控
- 对白是否自然
- 叙事是否连贯
- 可否局部重做

### 2.3 第三原则：分阶段生成，而不是一步直出
推荐路线：

构思 -> 小说圣经 -> 章节正文 -> 镜头脚本 -> 关键帧 -> 图生视频 -> 合成成片

而不是：

构思 -> 直接长视频

### 2.4 第四原则：用 I2V 作为视频主线
24GB 显存下，“图生视频（Image-to-Video）”明显比“纯文生视频（Text-to-Video）”更稳。  
原因：
- 角色一致性更好
- 构图更可控
- 镜头失败率更低
- 局部重做更方便

## 3. 最终推荐技术栈

## 3.1 LLM 主力：Qwen3-14B-Instruct-AWQ + vLLM
用途：
- 小说圣经生成
- 角色设定与章节大纲
- 小说正文
- 镜头脚本与台词拆分
- 视频提示词重写

选择原因：
- 14B 量级适合 24GB 单卡
- 中文能力强，适合小说写作
- 长上下文能力足够支撑“设定 + 已写章节 + 当前任务”
- 可以通过 vLLM 暴露 OpenAI 兼容 API，方便统一接入

不建议把 30B-A3B 作为常驻单卡主力，因为 24GB 单卡下会明显更紧张，调度难度更高。

## 3.2 多人 TTS 主力：Fish Audio S2 Pro
用途：
- 多角色对白
- 情绪化旁白
- 多说话人连续对话

选择原因：
- 支持多说话人
- 支持情绪/韵律标签
- 适合戏剧化对白
- 本地化价值高

## 3.3 中文旁白备选：CosyVoice 3.0
用途：
- 中文 narrator
- 发音精细控制
- 数字、专有名词、多音字处理

选择原因：
- 对中文生产问题更友好
- 支持 pronunciation inpainting
- 支持 text normalization

推荐策略：
- 默认对白：Fish
- 默认旁白：Fish 或 CosyVoice 二选一
- 中文发音要求高的片段：切到 CosyVoice

## 3.4 关键帧生成：Stable Diffusion 3.5 Medium + IP-Adapter + ControlNet
用途：
- 角色立绘
- 场景设定图
- 剧情关键帧
- 分镜画面

选择原因：
- 本地好部署
- 生态成熟
- 适合 ComfyUI 编排
- 可结合 IP-Adapter 做角色一致性
- 可结合 ControlNet 做构图/姿态/景别控制

## 3.5 视频生成：HunyuanVideo-1.5（优先 480p I2V step-distill）
用途：
- 将关键帧转为 3~5 秒动态镜头
- 将同一场景拆成多个短 shot 输出

选择原因：
- 24GB 单卡更现实
- 官方已有 I2V、step-distilled、加速路径
- 与“关键帧驱动镜头动画”的方案天然匹配

## 3.6 工作流与节点系统：ComfyUI
用途：
- 图像、视频相关节点式编排
- 统一管理 SD / ControlNet / IP-Adapter / HunyuanVideo
- 方便保存 workflow、复现结果、局部调试

## 3.7 质量检查：SenseVoice
用途：
- 对 TTS 回听转写
- 校验漏字、错字、发音异常
- 可做简单情绪识别辅助 QC

## 3.8 合成与后处理：FFmpeg
用途：
- 音频混音
- 视频裁切与拼接
- 字幕烧录
- 转场与最终导出

## 4. 系统总体架构

系统采用“编排器 + 多 worker”架构，但**不是多个大模型同时常驻 GPU**，而是通过 GPU 队列串行占用。

### 4.1 服务划分

1. orchestrator  
   - 负责任务调度
   - 负责编排全流程
   - 负责状态机与恢复

2. llm_worker  
   - Qwen3-14B-AWQ
   - 暴露本地 OpenAI 兼容接口

3. tts_worker  
   - Fish Speech / CosyVoice
   - 负责单句或批量配音

4. image_worker  
   - SD3.5 + IP-Adapter + ControlNet
   - 负责角色设定图与关键帧

5. video_worker  
   - HunyuanVideo-1.5 I2V
   - 负责短镜头生成

6. qc_worker  
   - SenseVoice
   - 对音频做 ASR 回查

7. compose_worker  
   - FFmpeg
   - 合成最终输出

### 4.2 GPU 调度原则

24GB 单卡不能同时舒服地承载：
- LLM 长上下文推理
- 图像扩散
- 视频扩散

所以推荐：
- 任何时刻只有一个“重模型”占 GPU
- 其他任务在 CPU 或空闲状态等待
- orchestrator 维护一个 GPU lock / job queue

### 4.3 推荐运行模式

顺序运行：
1. LLM 生成文本资产
2. TTS 生成音频
3. 图像生成关键帧
4. 视频生成镜头
5. FFmpeg 合成

如果要提高吞吐，可做“批次化串行”：
- 先写完一章所有镜头脚本
- 再批量 TTS
- 再批量关键帧
- 再批量镜头视频

## 5. 从构思到成片的完整流程

## 5.1 输入阶段：用户提交小说构思

### 输入建议结构
- 故事概述
- 风格要求
- 目标篇幅
- 视频风格
- 受众
- 是否要多人对白
- 是否要旁白
- 节奏偏好（快/慢）
- 画风（写实/动漫/国风/赛博）

### 输入 JSON 示例
```json
{
  "project_id": "novel_vid_001",
  "concept": "一位失忆的女剑客在边城酒馆醒来，被三方势力追杀。",
  "novel_style": "中文网文，偏电影感，阴郁克制",
  "target_novel_length": "第一卷8章",
  "target_episode": "第一章改编为8-12分钟视频",
  "visual_style": "写实电影感，冷色调，低饱和",
  "narration_mode": "旁白+对白",
  "audience": "18-35岁中文网文受众"
}
```

---

## 5.2 第一阶段：LLM 生成“小说圣经”

这是整个项目的核心基础层。  
不要一上来直接写正文，先生成以下中间资产：

### 产物 1：story_bible.json
字段建议：
- 世界观概要
- 故事主线
- 故事副线
- 时间线
- 阵营设定
- 禁止冲突的关键设定
- 主题与情绪曲线

### 产物 2：character_bible.json
字段建议：
- 角色名
- 年龄区间
- 外貌标签
- 身体特征
- 声线特征
- 口头禅
- 行为习惯
- 人物关系
- 视觉锚点
- 禁止偏移项

### 产物 3：episode_outline.json
字段建议：
- 每章标题
- 每章梗概
- 每章冲突点
- 每章情绪目标
- 每章 cliffhanger

### 产物 4：style_bible.json
字段建议：
- 叙述视角
- 句长风格
- 对白风格
- 节奏
- 画面感关键词
- 禁止使用的词汇或表达

### 生成提示词建议
LLM 任务不要只写“帮我写小说”，而应拆成多个明确子任务：
- 世界观整理
- 角色设定
- 章节拆分
- 风格守则
- 禁止冲突规则

---

## 5.3 第二阶段：LLM 生成小说正文

输入：
- 构思
- story_bible
- character_bible
- style_bible
- 当前章节 outline
- 已完成章节摘要

输出：
- chapter_01.md
- chapter_01_summary.json
- chapter_01_entities.json

### 正文生成原则
1. 每章完成后，强制做“章节摘要”
2. 提取新增角色、物品、地点、伏笔
3. 把这些变化写入长期记忆库

### 为什么要有章节摘要
因为本地模型虽然有长上下文，但你不应该每次都把全部正文塞进去。  
推荐用“全量设定 + 前文摘要 + 当前任务”来控制上下文长度。

---

## 5.4 第三阶段：从小说正文到镜头脚本

这是“文学文本”向“生产脚本”转换的关键步骤。

### 输出目标
把一章正文改写成：
- scene 列表
- shot 列表
- narrator 段落
- dialogue 段落
- 画面关键词
- 动作关键词
- 时长预估
- 音效建议

### 推荐数据结构：script.jsonl
每一行一个镜头或语音片段。

示例：
```json
{
  "scene_id": "SC01",
  "shot_id": "SC01_SH03",
  "role": "dialogue",
  "speaker": "女剑客",
  "text": "我不记得自己是谁，但这不是我的血。",
  "emotion": "restrained, cold, low voice",
  "visual_prompt": "close-up, blood on sleeve, dim tavern, candle flicker, cinematic realism",
  "motion_prompt": "slow push-in, subtle breathing, glance downward then lift eyes",
  "duration_hint_sec": 4.0,
  "camera": "close-up",
  "sfx": ["candle crackle", "wood creak"]
}
```

### 拆镜原则
- 单镜头建议 2~6 秒
- 对白驱动镜头建议 2.5~5 秒
- 旁白过场镜头建议 3~5 秒
- 超过 6 秒建议拆 shot

### 为什么这样拆
因为当前本地视频模型更擅长“短镜头稳定输出”，而不是“一个镜头承载大量复杂表演”。

---

## 5.5 第四阶段：语音层（多人配音）

### 4.1 先建立 voice_bible.json

每个角色一份声音定义：
- engine
- ref_audio
- style_tags
- speed
- pitch
- energy
- pause style
- language

示例：
```json
{
  "narrator": {
    "engine": "cosyvoice",
    "ref_audio": "voices/narrator.wav",
    "style_tags": ["calm", "cinematic", "controlled"],
    "speed": 1.0
  },
  "女剑客": {
    "engine": "fish",
    "ref_audio": "voices/swordswoman.wav",
    "style_tags": ["low", "cold", "restrained"],
    "speed": 0.96
  },
  "酒馆老板": {
    "engine": "fish",
    "ref_audio": "voices/innkeeper.wav",
    "style_tags": ["rough", "casual", "older"],
    "speed": 1.03
  }
}
```

### 4.2 TTS 任务拆分
- 旁白与对白分开渲染
- 每句单独生成 wav
- 再拼接成场景级音频
- 再拼接成整集音频

### 4.3 TTS 质量策略
1. 逐句生成，失败可重试
2. 每句保留原始文本与合成参数
3. 对专有名词、数字、地名可加“发音修正词典”
4. 关键台词允许人工标注重读/停顿

### 4.4 TTS 后处理
- loudness normalization
- 去静音头尾
- 根据角色风格调轻微 EQ
- 混入房间混响或环境感（可选）

---

## 5.6 第五阶段：关键帧层

视频的质量，很大程度上取决于关键帧阶段，而不只是视频模型本身。

### 5.1 关键帧分三类

#### A. 角色设定图
用途：
- 统一角色长相
- 作为 IP-Adapter 参考图

建议每个角色至少生成：
- 正面半身
- 侧面半身
- 全身
- 表情变化
- 不同光照下版本

#### B. 场景设定图
用途：
- 建立酒馆、街巷、雪夜、荒原、府邸等背景视觉基准

#### C. 剧情关键帧
用途：
- 对应每个 shot 的起始画面

### 5.2 角色一致性方案
推荐组合：
- SD3.5 Medium 生成 base image
- IP-Adapter 锁定角色参考图
- ControlNet 控制姿态/景别/构图
- 统一角色锚点词

角色锚点词示例：
- 黑色短发
- 左眼下旧伤
- 苍白肤色
- 暗红色内衬
- 旧式细剑
- 冷峻克制

### 5.3 关键帧生成流程
1. 先生成角色库
2. 再生成场景库
3. 最后按 shot 批量生成剧情关键帧

### 5.4 建议存档方式
每张关键帧都保存：
- prompt
- negative prompt
- seed
- model
- reference image list
- control strength
- ip-adapter scale
- output path

---

## 5.7 第六阶段：图生视频层

### 6.1 视频生成主策略
关键帧 + motion prompt -> HunyuanVideo-1.5 I2V -> 3~5 秒镜头

### 6.2 推荐初始参数
- 分辨率：480p
- 帧率：24fps
- 时长：3~5 秒
- 一章先做短镜头拼接
- 不要一开始追求长镜头

### 6.3 motion prompt 的写法
不要写小说原文，而要写摄影语言与动作语言。

错误示例：
“她想起了一切，心里突然很痛。”

正确示例：
“Slow push-in. She freezes for half a second, fingers tighten on the cup, breath becomes shallow, eyes lose focus briefly, candlelight flickers.”

### 6.4 镜头类型建议
- close-up：情绪
- medium shot：对白
- wide shot：环境建立
- insert shot：细节物件
- over-shoulder：对话关系

### 6.5 长镜头处理方式
不要强行单镜头做 10 秒戏剧表演。  
建议改为：
- 镜头 A：建立环境
- 镜头 B：人物反应 close-up
- 镜头 C：对白中景
- 镜头 D：物件细节

---

## 5.8 第七阶段：合成层

### 7.1 先以语音时长为基准
推荐方法：
- 每句对白/旁白先有明确时长
- 再选择对应镜头长度
- 视频略长则裁切
- 视频略短则补静帧、轻微慢放、插入过场镜头

### 7.2 音轨结构
建议至少分 4 条：
1. narrator bus
2. dialogue bus
3. sfx bus
4. music bus

### 7.3 字幕
每句对白和旁白都应自动生成字幕片段：
- start_time
- end_time
- speaker
- text

推荐输出：
- .srt（通用）
- .ass（样式更强）

### 7.4 FFmpeg 合成任务
- 拼接 shot
- 叠加字幕
- 混音
- 导出预览版
- 导出正式版

---

## 5.9 第八阶段：质量检查与回滚

### QC 1：TTS 回听转写
用 SenseVoice 把生成好的对白再次识别成文字，与原文比对：
- 是否漏字
- 是否错读
- 是否断句异常
- 是否情绪错位

### QC 2：角色一致性
用简单图像检索或 embedding 检查：
- 同一角色脸部差异过大
- 服装突然改变
- 发色或伤痕消失

### QC 3：镜头连续性
检查相邻镜头：
- 光线是否冲突
- 左右方向是否错
- 道具位置是否错

### QC 4：叙事节奏
检查：
- 旁白是否过密
- 对白是否太长
- 视频是否出现“音频已经结束但画面还在发呆”

### 回滚策略
系统必须支持局部重跑：
- 重跑一句 TTS
- 重跑一张关键帧
- 重跑一个 shot 视频
- 重做一个 scene 的拼接
- 不应要求全量重做一整集

## 7.2 Worker 任务接口建议

### llm_worker
输入：
- task_type

## 8. Prompt 设计建议

## 8.1 小说圣经提示词模板
目标：
- 生成高一致性的长期设定
- 不写正文，只写规则

要求：
- 输出 JSON
- 列出禁止冲突项
- 每个角色包含视觉锚点和声线锚点

## 8.2 章节正文提示词模板
目标：
- 依据 bible 写某章正文
- 严格遵守角色行为逻辑
- 保持文风

要求：
- 同章内不能自相矛盾
- 章末制造钩子
- 生成章节摘要

## 8.3 镜头脚本提示词模板
目标：
- 把正文转成生产脚本
- 每个镜头 2~6 秒
- 明确 narrator/dialogue/camera/motion

要求：
- 输出 JSONL
- 禁止文学化废话
- 用可执行的镜头语言

## 8.4 视频提示词重写器
目标：
- 把 visual prompt + motion prompt 重写成更适合 I2V 的英文提示词

要求：
- 保留角色身份
- 保留镜头语言
- 不引入新物件
- 不改变情绪基调

## 9. 本地部署方案

## 9.1 推荐操作系统
- Ubuntu 22.04/24.04 优先
- Windows 也可，但调度与依赖更麻烦

## 9.2 建议硬件
最低可用：
- GPU：24GB VRAM
- RAM：64GB
- CPU：8 核以上
- SSD：2TB NVMe

更舒适：
- RAM：128GB
- SSD：4TB

### 为什么内存和硬盘重要
视频生成与中间资产非常占空间，且多个节点会产生大量缓存。

## 9.3 推荐部署形式
- Python 3.11
- uv 或 conda 管理环境
- 各 worker 独立 venv 或容器
- Redis 作为任务队列（可选）
- SQLite / Postgres 记录项目状态

## 10. 调度与资源管理设计

## 10.1 任务队列模型
推荐状态机：
- pending
- queued
- running
- success
- failed
- retryable
- skipped

## 10.2 GPU 锁
任何需要 GPU 的任务开始前，必须申请 GPU 锁：
- 成功则执行
- 失败则排队
- 执行结束释放锁

## 10.3 批处理策略
- 先批量 TTS
- 再批量关键帧
- 再批量 I2V
- 避免频繁切模型

## 10.4 缓存策略
相同输入参数应允许复用缓存：
- 同一句对白不要重复合成
- 同一关键帧不要重复生成
- 同一 shot 不要重复出片

## 11. 关键难点与解决策略

## 11.1 角色一致性难
解决方案：
- 角色设定图先行
- IP-Adapter 固定参考图
- 角色锚点词固定
- 每个角色维护 face pack

## 11.2 文生视频随机性大
解决方案：
- 用 I2V 替代纯 T2V
- shot 拆短
- 固定 seed 与 reference
- 失败镜头可重试多次选择最优

## 11.3 中文 TTS 发音不稳定
解决方案：
- 专有名词字典
- 数字规范化
- 关键句走 CosyVoice
- TTS 后强制 ASR 回查

## 11.4 叙事节奏容易拖
解决方案：
- 旁白压缩
- 多用反应镜头与过场镜头
- 台词不要整段长句
- 一句台词对一个明确镜头动作

## 11.5 单卡切换模型慢
解决方案：
- 按阶段批处理
- 尽量少切
- 使用缓存
- 夜间批跑视频

## 14. 成本与产能预估（单机）

这个系统更适合：
- 单人作者
- 2~5 人小团队
- 长期迭代的本地创作工作室

它不适合：
- 一上来追求 4K 电影级长片
- 全自动零人工修正
- 单章几十分钟高动作复杂镜头的纯本地快速批量生产

最现实的成片形态是：
- 动态漫画
- 有声绘本
- 小说 PV
- 章节预告片
- 中短篇叙事视频

## 15. 最终推荐落地方案（拍板版）

### 主线配置
- LLM：Qwen3-14B-Instruct-AWQ
- LLM 服务：vLLM
- 多人对白：Fish Audio S2 Pro
- 中文旁白备选：CosyVoice 3.0
- 图像：Stable Diffusion 3.5 Medium
- 一致性：IP-Adapter + ControlNet
- 视频：HunyuanVideo-1.5 480p I2V step-distill
- 编排：ComfyUI + Python FastAPI orchestrator
- QC：SenseVoice
- 合成：FFmpeg

### 生产策略
- 一章一章做
- 一章拆 scene
- 一 scene 拆多个短 shot
- 先出音频，再对齐视频
- 支持局部重做

### 目标产物
第一阶段先做：
- 8~12 分钟一集
- 偏动态分镜
- 强旁白 + 强对白
- 稳定角色 + 稳定叙事

这是在 24GB 单卡上最现实、最稳、最容易迭代成功的路线。

## 16. 风险提示

1. 完全自动生成长篇高质量连续剧，目前依然不现实。  
2. 24GB 单卡可行，但需要严格调度，不要多个重模型同时常驻。  
3. 视频部分的随机性和失败率会高于文本与 TTS。  
4. 角色一致性是工程问题，不是单靠 prompt 能彻底解决。  
5. 真正好看的成片，通常仍需要人工挑选镜头与少量修稿。
